# برومت تعديل نظام التصنيفات — من مستويين إلى 5 مستويات هرمية

---

## المطلوب

تعديل نظام التصنيفات الحالي في مشروع مركز بيانات المنتجات (PDC) بحيث يدعم **حتى 5 مستويات هرمية** بدلاً من المستويين الحاليين (رئيسي + فرعي). الهدف هو توافق النظام مع هيكل التصنيفات في SAP.

---

## السياق التقني للمشروع

- **Backend:** Django 5 + Django REST Framework
- **Frontend:** React 18 + TypeScript + Vite
- **Database:** PostgreSQL (Neon)
- **Django App:** `categories` (موجود حالياً)
- **الوضع الحالي:** موديل Category فيه مستوى رئيسي (parent) ومستوى فرعي واحد (subcategory)
- **الوضع المطلوب:** موديل واحد self-referencing يدعم حتى 5 مستويات (Level 1 → Level 2 → Level 3 → Level 4 → Level 5)
- **صفحة إدارة التصنيفات:** موجودة حالياً في الـ Frontend — تحتاج تعديل لدعم الهيكل الجديد
- **الثيم:** النظام يدعم Dark/Light mode عبر CSS variables — التزم بالمتغيرات الموجودة
- **الخط:** 29LT Bukra (عربي) — الاتجاه RTL
- **اللغة:** الواجهة بالعربية

## التعديلات المطلوبة — Backend

### 1. تعديل موديل Category

عدّل الموديل الحالي ليصبح self-referencing مع دعم حتى 5 مستويات:

```python
class Category(models.Model):
    code = models.CharField(max_length=50, unique=True)  # كود التصنيف (مثل SAP)
    name_ar = models.CharField(max_length=255)  # الاسم بالعربية
    name_en = models.CharField(max_length=255, blank=True, null=True)  # الاسم بالإنجليزية
    parent = models.ForeignKey('self', on_delete=models.CASCADE, null=True, blank=True, related_name='children')
    level = models.PositiveIntegerField(default=1)  # 1 إلى 5
    sort_order = models.PositiveIntegerField(default=0)  # ترتيب العرض
    is_active = models.BooleanField(default=True)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ['sort_order', 'name_ar']
        verbose_name = 'تصنيف'
        verbose_name_plural = 'تصنيفات'
```

**قواعد مهمة:**
- الحد الأقصى 5 مستويات — لا يُسمح بإضافة تصنيف بمستوى 6
- الـ level يتحسب تلقائياً من الـ parent (parent=None يعني level=1)
- عند حذف تصنيف أب، تُحذف كل الفروع تحته (CASCADE)
- الـ code يكون unique ومطلوب
- مو شرط كل فرع يكون 5 مستويات — بعض الفروع ممكن تكون 2 أو 3 مستويات فقط

### 2. تعديل الـ Serializers

```python
# Serializer يعرض الشجرة كاملة (recursive)
class CategoryTreeSerializer(serializers.ModelSerializer):
    children = serializers.SerializerMethodField()
    products_count = serializers.IntegerField(read_only=True)  # عدد المنتجات في هذا التصنيف

    class Meta:
        model = Category
        fields = ['id', 'code', 'name_ar', 'name_en', 'level', 'parent', 'is_active', 'sort_order', 'children', 'products_count']

    def get_children(self, obj):
        children = obj.children.filter(is_active=True).order_by('sort_order', 'name_ar')
        return CategoryTreeSerializer(children, many=True).data

# Serializer للإضافة والتعديل (flat)
class CategoryCreateUpdateSerializer(serializers.ModelSerializer):
    class Meta:
        model = Category
        fields = ['code', 'name_ar', 'name_en', 'parent', 'is_active', 'sort_order']
```

### 3. تعديل الـ Views / API Endpoints

```
GET    /api/categories/tree/          → شجرة التصنيفات كاملة (nested)
GET    /api/categories/flat/          → كل التصنيفات flat list (للقوائم المنسدلة)
GET    /api/categories/{id}/          → تصنيف واحد مع أبنائه المباشرين
POST   /api/categories/              → إضافة تصنيف جديد
PUT    /api/categories/{id}/          → تعديل تصنيف
DELETE /api/categories/{id}/          → حذف تصنيف (مع أبنائه)
POST   /api/categories/import-excel/  → رفع ملف Excel للتصنيفات
GET    /api/categories/{id}/path/     → مسار التصنيف من الجذر للورقة (breadcrumb)
```

### 4. رفع التصنيفات عبر Excel

أضف endpoint يقبل ملف Excel بالأعمدة التالية:
- `code` — كود التصنيف (مطلوب، unique)
- `name_ar` — الاسم بالعربية (مطلوب)
- `name_en` — الاسم بالإنجليزية (اختياري)
- `parent_code` — كود التصنيف الأب (فارغ = تصنيف رئيسي)
- `sort_order` — ترتيب العرض

النظام يقرأ الملف ويبني الشجرة تلقائياً. يتحقق من:
- عدم تجاوز 5 مستويات
- عدم تكرار الأكواد
- وجود الـ parent_code في النظام أو في نفس الملف

### 5. Migration

اكتب migration يحوّل البيانات الحالية (المستويين) للهيكل الجديد بدون فقدان بيانات.

## التعديلات المطلوبة — Frontend

### 1. صفحة إدارة التصنيفات

عدّل الصفحة الحالية لتدعم عرض وإدارة 5 مستويات:

**عرض الشجرة (Tree View):**
- عرض التصنيفات كشجرة هرمية قابلة للطي/الفتح (expandable/collapsible)
- كل مستوى يكون بإزاحة (indent) واضحة عن المستوى الأب
- أيقونة سهم (▶/▼) للتصنيفات اللي عندها أبناء
- عرض الكود والاسم وعدد المنتجات لكل تصنيف
- عرض الـ level كـ badge ملون (Level 1 لون مختلف عن Level 2 وهكذا)
- زر إضافة تصنيف فرعي يظهر عند hover على أي تصنيف (بشرط ما يتجاوز 5 مستويات)

**إضافة/تعديل تصنيف (Modal أو Drawer):**
- حقل الكود (code) — مطلوب
- حقل الاسم بالعربية (name_ar) — مطلوب
- حقل الاسم بالإنجليزية (name_en) — اختياري
- قائمة منسدلة لاختيار التصنيف الأب (parent) — تعرض المسار الكامل مثلاً: "سيراميك > أرضيات > 60×60"
- الـ level يتحسب تلقائياً ويعرض للمستخدم
- تحقق: إذا التصنيف الأب المختار مستوى 5، لا يسمح بالإضافة ويعرض رسالة "الحد الأقصى 5 مستويات"

**حذف تصنيف:**
- تأكيد قبل الحذف مع تنبيه: "سيتم حذف هذا التصنيف وجميع التصنيفات الفرعية تحته"
- إذا فيه منتجات مرتبطة، يعرض تحذير إضافي

**رفع Excel:**
- زر "رفع تصنيفات من Excel" في أعلى الصفحة
- يقبل .xlsx و .xls
- يعرض ملخص قبل الحفظ: "سيتم إضافة X تصنيف جديد"
- يعرض الأخطاء إن وجدت (أكواد مكررة، تجاوز 5 مستويات)

**Breadcrumb:**
- عند اختيار تصنيف، يعرض المسار الكامل في أعلى الصفحة
- مثال: الرئيسية > سيراميك > أرضيات > 60×60

### 2. تعديل صفحات المنتجات

في كل مكان يظهر فيه اختيار التصنيف (إضافة منتج، تعديل منتج، فلاتر الكتالوج):
- غيّر القائمة المنسدلة العادية إلى قائمة هرمية (cascading dropdown أو tree select)
- المستخدم يختار من الشجرة ويمكنه اختيار أي مستوى
- يعرض المسار الكامل بعد الاختيار

### 3. فلاتر الكتالوج

- عدّل فلاتر صفحة الكتالوج لتدعم التصفية حسب أي مستوى
- عند اختيار تصنيف أب، يعرض كل المنتجات في جميع التصنيفات الفرعية تحته

## قواعد عامة

1. **التزم بالثيم الحالي** — استخدم CSS variables الموجودة (--color-bg, --color-surface, --color-text-primary, --color-gold, إلخ)
2. **RTL** — كل شي من اليمين لليسار
3. **Responsive** — يشتغل على الموبايل والديسكتوب
4. **لا تحذف أي كود حالي** بدون ما تكون متأكد إنه مو مستخدم
5. **اكتب migration آمنة** — لا تفقد بيانات التصنيفات الحالية
6. **الأداء** — استخدم select_related و prefetch_related للـ queries اللي تجلب الشجرة
7. **Validation** — تأكد إن الـ level ما يتجاوز 5 في Backend و Frontend
